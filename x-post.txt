rvLLM -- vLLM rewritten in Rust. One dev, $1,780.
6,098 tok/s (N=32) on A100. 218 tok/s single-sequence (4.5ms/tok). near-linear scaling to N=16. 20x faster startup. 31x smaller.

what is it
from-scratch Rust implementation of vLLM. drop-in OpenAI API replacement. same endpoints, same format, just point your client at the Rust server. 16 MB binary.

why
no GIL, no GC, no PyTorch. GPU-side argmax (512 bytes PCIe vs 74 MB logit transfer). deterministic memory via Rust ownership. single static binary vs ~500 MB pip install.

architecture
- 23 Rust crates, strict dependency tree from GPU primitives to HTTP API
- 15 hand-written CUDA kernels compiled to PTX (sm_70 through sm_120)
- full f16 forward path -- zero cast kernels in the decode inner loop
- fused QKV + gate+up weight concatenation (5 GEMMs -> 2 per layer)
- cross-layer residual+RMSNorm fusion (-28 kernel launches per forward)
- CUDA graph capture/replay (full decode forward as single graph launch)
- dedicated GPU thread (tokio async loop stays responsive during 4ms compute)
- async DtoH with pinned host memory
- FlashAttention-2 with paged KV cache, GQA, f16 I/O
- cuBLAS hgemm + cublasLt split-K for decode GEMMs
- f16 KV cache with paged block allocation
- continuous batching with preemption and prefix caching
- 10 model architectures (Llama, Mistral, Qwen2, Phi, Gemma, Mixtral, DeepSeek, GPT-NeoX, StableLM, Cohere)
- 792 tests across 23 crates

verified throughput (A100 80GB SXM4, Qwen2.5-1.5B, greedy, 32 tok/req)

system comparison
- startup: 6s vs ~120s (20x faster)
- binary: 16 MB vs ~500 MB (31x smaller)
- CPU RSS: 348 MB vs ~1 GB (3x less)

optimization journey (N=1 single-sequence tok/s)
phase 4: 130 -- CUDA graph capture (fixed 3 root causes)
phase 5: 174 -- 10-agent swarm (cast reduction, fused ops, engine opt)
full f16: 200 -- zero casts, all f16 kernels
9-agent:  236 -- cross-layer fusion, memset elimination, pool tuning
GPU thread: 218 -- dedicated OS thread, peak N=32 jumps to 6,098

the 4.1ms per token at N=1 is genuine GPU memory bandwidth time (3GB weights on A100 @ 2039 GB/s). custom GEMV = cuBLAS = cublasLt = same speed. the kernel efficiency frontier.

next: INT8/FP8 quantization (halve weight reads), speculative decoding, async engine overlap.
